# TravelAI Center 设计标准

## 1. 项目定位

TravelAI Center 是一个文旅经营中枢 demo，目标是基于景区、酒店、票务、商业、停车、客源等数据，做出一个更有判断力和作品感的前端产品。

它不是传统后台管理系统，也不是黑底数据大屏。它应该像一个冷色、克制、有高级感的经营工作台：首屏先告诉用户今天经营是否正常、哪里有风险、下一步应该做什么，再让用户进入各业务面板查看详情。

## 2. 技术方向

- 前端框架：Vue 3 + Vite + TypeScript
- 样式：Tailwind CSS
- UI 基础组件：shadcn-vue + Reka UI
- 图表：ECharts
- 路由：Vue Router
- 状态：Pinia
- 数据：第一版使用本地 mock 数据，后续预留 Spring Boot API 适配层
- 主题：使用 CSS Variables 做整站冷色主题切换，默认主题为“暗夜模式”

## 3. 第一版页面范围

第一版先做 4 个页面，不做 12 个页面全量复刻。

- 经营首页：经营状态、异常队列、经营链路、核心 KPI、今日建议
- 实时监控：酒店、票务、商业、停车、景区项目面板
- 预售预测：未来日期热力、风险日期、预售收入、出租率
- 复盘报告：自动摘要、数据证据、行动项、经营结论

移动端不是完整后台缩小版，而是经营简报版：底部导航、卡片纵排、重点展示状态、异常和行动。

## 4. 竞品风格点评与避让

竞品西软 XDMP 的优点是数据覆盖完整，酒店、票务、商业、停车、客源等模块都齐，适合查数和做运营看板。

但它的问题也明显：页面偏传统 BI 和后台系统，长页面多，内部滚动多，宽表多，图表和指标堆叠较重。用户能看到很多数据，但需要自己判断重点，缺少异常解释、经营结论和动作闭环。

本项目可以对标它的功能面板，但风格不能雷同。我们不做蓝色传统后台，不做黑底大屏，不做图表堆满屏。我们做“经营判断优先”的文旅工作台。

## 5. 总体视觉原则

- 冷色、清晰、克制，不压抑、不炫技。
- 现代 SaaS 工作台质感，但保留文旅行业的温度和一点拼色记忆点。
- 页面应该有呼吸感，但不能空泛。
- 信息密度适中，重点明确。
- 首屏要像产品，不像模板。
- 所有页面都要同时考虑 PC 和移动端。

## 6. 明确审美约束

这些约束优先级很高，所有页面都必须遵守：

- 所有文字，尤其是标题，不要过粗。
- 小字不要显得太小，辅助文字也要可读。
- 避免使用全大写英文单词。
- 避免使用太多卡片，页面要有整块区域和横向信息带。
- 尽量避免使用阴影。
- 尽量避免使用状态圆点。
- 白背景上避免太淡的灰色，要提高对比度和可见性。
- 红色不要鲜亮刺眼，风险色要克制。
- 避免卡片套卡片。
- 金额数字和标题可以使用更有风格的常见 UI 字体，但不能牺牲可读性。

## 7. 字体规范

推荐字体栈：

```css
font-family: Inter, "SF Pro Display", "SF Pro Text", "PingFang SC", "Microsoft YaHei", system-ui, sans-serif;
```

数字和金额可使用：

```css
font-family: "DIN Alternate", "Inter", "SF Pro Display", "PingFang SC", system-ui, sans-serif;
```

标题不要使用 `font-bold` 或更重字重。建议：

- 页面标题：`font-medium` 或 `font-semibold`，字号 24-32
- 区块标题：`font-medium`，字号 16-20
- 卡片指标标题：`font-medium`，字号 14-16
- 正文：字号 14-16
- 辅助文字：字号不小于 13

移动端正文不要低于 14px，辅助文字不要低于 13px。

## 8. 色彩规范

使用冷色专业工作台风格。默认主题为深蓝黑暗色模式，同时保留明亮冷色主题用于对比。拼色只用于区分经营模块和形成产品记忆点，不能变成花哨装饰。

基础色：

- 暗色背景：`#040810`、`#07111f`、`#0b1626`
- 暗色面板：`#09121f`、`#0c192a`
- 明亮冷色背景：`#eef7ff`、`#f8fbff`
- 主文字：暗色主题使用 `#f1f8ff`，亮色主题使用 `#111f2d`
- 次级文字：暗色主题使用 `#94a7be`，亮色主题使用 `#3d4f63`
- 边框：使用低饱和蓝灰，不使用暖灰、米色或棕色

业务语义色：

- 冷青：`#4dd3e1`
- 科技蓝：`#60a5fa`
- 极光紫：`#a48eff`
- 克制红：`#f4727e`
- 柔金黄：`#eec15b`
- 深墨蓝：`#040810`

不要使用鲜红、纯蓝、纯紫作为大面积主色。背景允许冷色光感，但不要做黑底大屏式炫光。暗色主题要保证可读性和层次，不能压抑。

## 8.1 主题系统

项目保留多主题切换，用于快速比较视觉方向。主题切换器放在 PC 侧边栏和移动端底部导航上方，切换后写入 `localStorage`。

主题列表：

- 极光冷蓝：蓝白机甲感，白底、强蓝边框和少量红黄点缀，偏传统科技产品。
- 暗夜模式：默认主题，深蓝黑、高对比、沉浸。
- 冰川白：明亮冷调、干净专业。
- 石墨青：克制、耐看、偏工具感。

主题必须一起影响：页面背景、面板背景、边框、状态标签、导航、信息块、ECharts 图表色盘。

## 9. 布局规范

PC 端：

- 左侧导航保持克制，宽度 220-248px。
- 顶部使用项目状态条，而不是传统厚重 header。
- 首屏结构建议为：经营状态条 + 异常/建议 + 经营链路 + 核心指标。
- 页面区块尽量使用全宽工作区，不要把每个区域都做成卡片。
- 卡片只用于单个指标、单个行动项、单个重复列表项。

移动端：

- 使用底部导航，不强行保留 PC 左侧导航。
- 页面内容纵向排列，先显示状态、异常、行动，再显示图表。
- 避免复杂宽表，宽表改为摘要列表或可横向滚动的明细区域。

响应式断点：

- `sm`：640px
- `md`：768px
- `lg`：1024px
- `xl`：1280px

## 10. 组件规范

建议组件：

- `AppShell`：整体布局
- `SideNav`：PC 侧边导航
- `MobileTabBar`：移动端底部导航
- `ProjectStatusBar`：项目状态条
- `MetricStrip`：核心指标条
- `MetricCard`：单个指标卡
- `RiskQueue`：异常队列
- `ActionList`：行动项列表
- `JourneyFlow`：经营链路
- `TrendPanel`：趋势面板
- `ForecastCalendar`：预售预测日历
- `ReportSummary`：复盘摘要
- `EvidenceList`：数据证据列表

组件设计原则：

- 不要卡片套卡片。
- 不要为了装饰添加多层阴影。
- 按功能拆组件，不要按视觉碎片过度拆分。
- 每个组件的数据结构要清晰，方便以后接后端接口。

## 11. 图表规范

图表统一使用 ECharts 实现，用于回答问题，不用于填满页面。图表要体现“前端有经验”：留白、坐标轴、tooltip、颜色、响应式都要处理干净。

优先使用：

- 折线图：分时趋势、收入趋势、客流趋势
- 条形图：排行、渠道贡献、商品销量
- 日历热力图：预售、节假日、风险日期
- 漏斗图或链路图：停车、入园、游玩、消费转化
- 地图：客源省市分布

图表颜色要克制但不暗沉，避免彩虹色。建议使用冷青、科技蓝、极光紫、柔金黄、克制红组成 3-5 色图表系统。图例和坐标轴文字要清晰，不要太淡。

ECharts 细节要求：

- 折线图使用柔和面积渐变，但透明度要低。
- 柱状图使用圆角柱，颜色按业务模块区分。
- tooltip 跟随主题，暗色主题使用深色面板，亮色主题使用浅色面板，边框清晰、阴影克制。
- grid 留足边距，避免标签被裁切。
- 移动端图表高度不要过高，优先展示趋势判断。

## 12. 页面内容规范

不要只展示数据，要展示经营判断。

每个核心页面都应尽量包含：

- 当前状态
- 关键指标
- 异常或机会
- 原因候选
- 建议动作
- 可下钻的证据

示例：

- 不只写“停车收入 ¥485”，还要写“停车收入低于昨日，主要受入场峰值提前影响”。
- 不只写“票务入园 709 人”，还要提示“13:00-15:00 可能出现核销压力”。
- 不只写“商业收入 ¥27”，还要提示“入园转化弱，需检查店铺开业状态或动线导流”。

## 13. Mock 数据规范

第一版使用本地 mock 数据，建议放在：

```text
src/data/mock-dashboard.ts
src/data/mock-monitor.ts
src/data/mock-forecast.ts
src/data/mock-report.ts
```

数据结构要模拟后端 API，可后续替换为 Spring Boot 接口。

命名示例：

```ts
type Metric = {
  id: string
  label: string
  value: string | number
  unit?: string
  trend?: number
  status: 'normal' | 'warning' | 'risk'
  insight?: string
}
```

## 14. 路由规范

建议路由：

- `/`：经营首页
- `/monitor`：实时监控
- `/forecast`：预售预测
- `/report`：复盘报告

页面标题使用中文，URL 使用英文小写。

## 15. 发布规划

推荐流程：

1. 本地初始化项目。
2. 推送到 GitHub 空仓库。
3. Vercel 绑定 GitHub 仓库。
4. Vercel 自动构建并发布前端。
5. 后续再接 Spring Boot 后端和 PostgreSQL 数据库。

Vercel 只承载前端。Spring Boot 后端不建议部署在 Vercel，可后续选择 Render、Fly.io、Railway 等平台。

GitHub 推送说明：

- GitHub 已不支持使用账号密码直接推送。
- SSH 推送需要本机 SSH key 已加入 GitHub。
- HTTPS 推送可以用 Personal Access Token，命令行提示输入 password 时粘贴 token。
- 不安装 `gh` CLI 也可以推送，只要在 GitHub 网页生成 token。
- 如果使用 HTTPS remote，可执行：

```bash
git remote set-url origin https://github.com/justwu/travelai-center.git
git push -u origin main
```

随后 username 输入 GitHub 用户名，password 粘贴 Personal Access Token。

## 16. 给另一个 AI 的初稿任务

请先阅读本文件，再基于本文件生成 TravelAI Center 的前端初稿。

要求：

- 技术栈使用 Vue 3 + Vite + TypeScript + Tailwind CSS + shadcn-vue + Reka UI + ECharts。
- 默认视觉使用“暗夜模式”主题，整体要冷色、清晰、有高级感，但不能压抑。
- 保留主题切换器，并提供极光冷蓝、暗夜模式、冰川白、石墨青四套主题。
- 先实现 4 个页面：经营首页、实时监控、预售预测、复盘报告。
- 页面风格必须与竞品西软 XDMP 有明显差异。
- 遵守第 6 节审美约束。
- PC 和移动端都要可访问。
- 移动端优先展示经营状态、异常、行动项，不强行塞宽表。
- 使用 mock 数据，不接真实后端。
- 完成后用浏览器 MCP 打开页面，分别检查 PC 和移动端效果，并截图留证。

重点检查：

- 标题是否过粗。
- 小字是否过小或太淡。
- 是否有卡片套卡片。
- 是否使用了过多阴影。
- 白底上的灰色对比度是否足够。
- 红色是否太鲜亮。
- 是否过度使用状态圆点。
- 页面是否像现代文旅经营产品，而不是普通后台模板。
